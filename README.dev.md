# Unity Runtime Template (Developer Notes)

A Unity project template focused on **predictable runtime memory**, **low-GC gameplay code**, and **modular systems** you can reuse in upcoming games.

This repository is my baseline for future projects where I want to avoid ad-hoc allocations and build core gameplay features on top of reusable runtime primitives.

## Why this template exists

Most prototype code starts fast but becomes hard to scale: frequent GC spikes, container churn, and logic tied to scene scripts.

This template addresses that by:

- introducing allocator-backed containers early,
- using handle-based access patterns for safety,
- keeping systems in assembly-based modules,
- and providing a tween runtime as a real usage example.

## Why allocators and fixed collections?

### 1) Predictable memory usage

Runtime capacity is explicit (`capacity` in allocator configs), so memory growth is planned instead of accidental.

### 2) Lower GC pressure

Data is stored in preallocated arrays and accessed via handles/spans, which helps avoid per-frame heap allocations in hot paths.

### 3) Better runtime safety

Handles carry generation data, so stale references can be detected and rejected (`InvalidOperationException` on invalid handle state).

### 4) Faster iteration on systems code

With shared allocator + container primitives, new systems (AI queues, combat buffers, effect pools, etc.) can be built with consistent behavior.

### 5) Easier performance tuning

Capacity and allocator strategy (`Persist`, `Arena`, `Stack`) are clear tuning knobs depending on lifecycle:

- `Persist<T>`: long-lived data, no deallocation.
- `Arena<T>`: grouped allocations with clear/reset style usage.
- `Stack<T>`: LIFO-friendly allocation/deallocation pattern.

## What is included

- **Core** (`Assets/Module/Core`)
  - `Handle`, `HandleArray`, `Logger`
- **Allocator** (`Assets/Module/Allocator`)
  - `IAllocator<T>`, `Persist<T>`, `Arena<T>`, `Stack<T>`
- **Collection** (`Assets/Module/Collection`)
  - `FixedArray<T>`, `FixedList<T>`, `FixedQueue<T>`, `FixedStack<T>`
- **Tween Runtime** (`Assets/Module/Tween/Core` + `Assets/Module/Tween`)
  - dispatcher/state/manager + extension API (`DoMoveX`, `DoMoveY`, etc.)
- **Gameplay Example** (`Assets/Module/Gameplay`)
  - small script showing tween usage in coroutine flow

## Architecture notes

- Project uses asmdefs per module to keep dependencies explicit.
- `Glai.Allocator` and `Glai.Collection` allow unsafe code (for future low-level optimization work).
- Runtime currently targets Unity `6000.3.8f1` and C# 9 / `netstandard2.1` (Unity-generated project files).

## Quick start

### Prerequisites

- Unity Editor `6000.3.8f1`
- Windows PowerShell (examples below)

Set Unity executable path:

```powershell
$env:UNITY_EXE = "C:\Program Files\Unity\Hub\Editor\6000.3.8f1\Editor\Unity.exe"
```

### Compile check

```powershell
& $env:UNITY_EXE -batchmode -nographics -projectPath . -quit -logFile Logs/compile.log
```

### Run all EditMode tests

```powershell
& $env:UNITY_EXE -batchmode -nographics -projectPath . -runTests -testPlatform editmode -testResults Logs/editmode-results.xml -logFile Logs/editmode-tests.log
```

### Run a single test (important)

```powershell
& $env:UNITY_EXE -batchmode -nographics -projectPath . -runTests -testPlatform editmode -testFilter "MyGame.TweenTests.Dispatch_CompletesTween" -testResults Logs/single-test.xml -logFile Logs/single-test.log
```

Use fully-qualified name format:

`Namespace.ClassName.MethodName`

## Usage examples

### Allocator + fixed list

```csharp
var allocator = new Glai.Allocator.Persist<int>(new Glai.Allocator.PersistData
{
    name = "DamageBuffer",
    capacity = 1024
});

var values = new Glai.Collection.FixedList<int>(256, allocator);
values.Add(10);
values.Add(25);
```

### Tween API

```csharp
using Glai.Tween;

var handle = transform.DoMoveY(0f, 10f, 1.5f);
Tween.SetTweenSpeed(handle, 2.0f);
yield return handle.ToYield();
```
