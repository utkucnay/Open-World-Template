## Code Style Guidelines (Observed + Enforced)

### Formatting
- Use 4 spaces for indentation (no tabs).
- Use Allman brace style (`{` on new line for types/methods/blocks).
- Keep methods focused; prefer short guard clauses.
- Avoid unnecessary comments; use names and structure for clarity.

### Imports (`using`)
- Keep `using` directives at file top (outside namespace).
- Preferred order for new files:
  1) `System*`
  2) Unity namespaces (`UnityEngine`, `Unity.Mathematics`, etc.)
  3) project namespaces (`Glai.*`)
- Remove unused imports.

### Naming
- Types, methods, properties, enums: PascalCase.
- Private fields/local vars/parameters: camelCase.
- Existing code does **not** use `_camelCase` backing fields; follow local file style.
- Data-carrier structs (`PersistData`, `ArenaData`, etc.) currently use lowercase fields (`name`, `capacity`); maintain consistency in that area.

### Types and API Design
- Prefer explicit types for public API surface.
- Use `var` when RHS type is obvious and improves readability.
- Constrain generic value containers with `where T : struct` when required by allocator model.
- Prefer structs for lightweight handle/value concepts (`Handle`, `TweenHandle`, fixed collections).
- Keep allocator APIs span-based and allocation-conscious where possible.

### Error Handling
- Validate indices and handles before access.
- Throw specific exceptions already used by project patterns:
  - `ArgumentOutOfRangeException` for index bounds.
  - `InvalidOperationException` for invalid handle/state.
- Use clear exception messages that describe why the operation is invalid.
- Do not swallow exceptions silently.

### Unity-Specific Practices
- Runtime singletons should be initialized via Unity lifecycle attributes when needed.
- Keep engine-facing classes in appropriate asmdefs; avoid cross-module leakage.
- Avoid heavy per-frame allocations in `Update`/dispatch loops.
- Prefer deterministic container capacities for runtime systems.

### Performance and Allocation
- Favor fixed-size or pooled structures in hot paths.
- Avoid LINQ in performance-critical loops.
- Reuse handles and indices carefully; preserve generation checks for safety.

## Assembly Definition Guidance
- Keep dependencies minimal and explicit in `.asmdef` files.
- `Glai.Allocator` and `Glai.Collection` permit unsafe code; do not introduce unsafe in other modules unless justified.
- Place new code in the closest existing module instead of dumping into `Assembly-CSharp`.
